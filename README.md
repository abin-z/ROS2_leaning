# ROS2_leaning

简单来说，**ROS 2 是一个用于开发机器人系统的开源软件框架**。

你可以把它理解为机器人的“操作系统”，但它并不是像 Windows 或 Linux 那样直接管理硬件，而是提供了一套**通信、调度和工具**，让机器人的各个部件能够协同工作。

它是现代机器人开发的事实标准，被广泛应用于工业、自动驾驶、服务机器人等几乎所有领域，是进入机器人行业的重要技能。

为了帮你更好地理解，我们来拆解一下：

### 一、ROS 2 的核心：解决机器人开发中的通信问题

一个机器人（比如一个移动底盘）有很多传感器和部件：激光雷达、摄像头、电机、控制器等。ROS 2 的核心作用是让这些部件之间能够**高效、可靠地交换信息**。

它主要通过三种通信方式来实现，你可以对比成"微信群聊"、"打电话"和"下订单"：

| 通信方式           | 比喻       | 特点                                                         | 场景举例                                                     |
| :----------------- | :--------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **话题 (Topic)**   | 微信群聊   | **异步、持续性**。一方发布消息，所有订阅者都能收到。         | 摄像头持续发布“图像”消息，导航模块和显示界面都订阅这个“话题”来获取图像。 |
| **服务 (Service)** | 打电话     | **同步、一次性**。客户端发出请求，等待服务器回复。           | 节点A请求“计算路径”，节点B计算后回复“路径是[xxx]”。          |
| **动作 (Action)**  | 下外卖订单 | **长时间执行，可反馈**。发送目标，可中途取消，并持续收到进度反馈。 | 控制机械臂去抓取物体，可以一直收到“已移动到30%”的进度反馈。  |

> **一个形象的类比**：
> ROS 2 的通信机制很像**快递系统**。你可以把“传感器数据”或“控制指令”想象成一个个**快递包裹**。不同的通信方式，就是寄送这些包裹的不同方式。
>
> - **话题**就像**订阅报纸**：你订阅了，报纸（数据）就会定期送到你信箱，不用每次都打电话去要。
> - **服务**就像**发挂号信**：你寄出信后，会等待对方寄回一封回执，一来一回，当场办结。
> - **动作**就像**寄一个需要跟踪的快递**：你寄出后，会不断收到“已揽收”、“运输中”、“派送中”的进度通知，直到对方签收才算完成。

### 二、ROS 2 的组成：不仅是一套通信规则

除了通信，ROS 2 还提供了完整的开发生态：

1. **节点 (Node)**：一个可执行程序的最小单位。一个机器人可以有很多节点，比如一个“摄像头节点”、一个“激光雷达节点”和一个“控制节点”。
2. **包 (Package)**：ROS 2 软件的打包形式。你可以通过 `sudo apt install ros-humble-slam-toolbox` 这样的命令来安装别人写好的功能包（比如建图包、导航包）。
3. **分布式能力**：ROS 2 的节点可以不局限于同一台电脑。你可以让摄像头节点运行在树莓派上，而运行复杂算法的节点运行在性能更强的PC上，它们通过网络即可通信。
4. **丰富好用的工具**：
   - **`rqt`**：一个图形化调试工具集，可以查看所有节点和话题的实时通信状态。
   - **`rviz2`**：强大的3D可视化工具，可以显示机器人模型、激光雷达点云、摄像头图像等。
   - **`rosbag2`**：录制和回放工具。你可以先开车跑一圈，录制下所有传感器数据，然后回家在电脑上反复回放这些数据来调试算法，无需真车。

### 三、ROS 2 vs. ROS 1：为什么直接学新的ROS2？

ROS 1 在2010年左右发布，是开创性的，但也存在一些设计上的问题，比如**实时性差、分布式通信依赖一个中心网络节点（如果这个节点挂了整个系统就挂了）、对嵌入式支持不好**。

ROS 2 在2017年推出，完全重新设计，**解决了上述所有痛点**：

- 采用**DDS（数据分发服务）** 作为通信中间件，实现了真正的分布式、高可靠、支持实时通信。
- 去掉了**中心节点**，系统更健壮。
- 支持Windows、macOS，特别是对**嵌入式系统（如树莓派、STM32）** 的支持更好。

**官方已明确 ROS 1 将停止维护，所有新项目都应基于 ROS 2 开发。**



## ROS2工作空间

简单来说，**ROS 2 的工作空间（Workspace）就是一个文件夹**，你在这个文件夹里**下载、配置、编译和运行**你的 ROS 2 代码。它把所有相关的代码、编译产物和运行文件都集中在一个地方。

可以把它类比成一个**项目工程文件夹**，就像你用 Visual Studio 创建一个“解决方案”时，会生成一个包含所有项目文件的目录一样。

### 一、工作空间的典型结构

一个标准的 ROS 2 工作空间（比如你之前学习时可能创建的 `ros2_ws` 文件夹），其内部结构如下：

text

```
ros2_ws/                     <-- 这就是你的工作空间根目录
├── src/                     <-- 1. 源代码空间：存放所有功能包的源码
│   ├── package_a/
│   ├── package_b/
│   └── ...
├── build/                   <-- 2. 编译空间：存放编译过程中产生的临时文件
├── install/                 <-- 3. 安装空间：存放最终的可执行文件、库文件、脚本等
└── log/                     <-- 4. 日志空间：存放编译和运行时的日志信息
```



这4个目录各有各的用途：

| 目录           | 作用                                                         | 是否需要关注                                                 |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`src/`**     | **你自己写的** 或 **从网上下载** 的所有功能包（package）的源代码都放在这里。 | **是**。你的所有开发工作几乎都在这个文件夹下进行。           |
| **`build/`**   | 编译工具 `colcon` 在这里进行实际的编译工作，生成中间文件。   | **否**。你通常不需要管它，甚至可以随时删除，下次编译会自动生成。 |
| **`install/`** | **编译完成后的最终产物**。这里包含了可以直接运行的**可执行文件**、**库文件**、**配置文件**和**头文件**。你在终端运行的 `ros2 run <package> <executable>` 命令，实际上就是在 `install` 目录里找可执行文件。 | **否**。这是编译结果，你不需要手动修改，但要让系统知道它的存在（通过 `source` 命令）。 |
| **`log/`**     | 每次编译或运行节点时产生的**日志文件**存放在这里。           | **否**。只在需要调试编译或运行错误时才可能查看。             |

### 二、为什么需要工作空间？

工作空间的主要好处有3点：

1. **隔离环境，避免冲突**：你可以为不同的项目创建不同的工作空间。比如，你有一个用 ROS 2 Humble 的机器人项目A，又有一个用 ROS 2 Jazzy 的机器人项目B，它们可以在各自的 workspace 里独立开发，互不干扰。
2. **便于管理和重现**：整个项目（源代码 + 编译配置）都集中在一个文件夹里。你可以把这个 `ros2_ws` 文件夹打包发给同事，或者上传到 GitHub，他下载后就能直接编译运行，极大地提高了协作效率。
3. **不污染系统**：你自己的代码编译后都放在 `install` 目录下，而不会安装到系统的 `/opt/ros/humble` 目录里。这样既不需要 `sudo` 权限，也保证了系统 ROS 2 环境的纯净。

### 三、如何使用工作空间？（核心三步）

使用工作空间的流程，其实就是你每次开发都会重复做的三步：

1. **创建**：

   bash

   ```bash
   mkdir -p ~/ros2_ws/src   # -p 确保创建父目录
   cd ~/ros2_ws
   ```

   

2. **编译**：进入工作空间根目录，运行 `colcon build`。

   bash

   ```bash
   cd ~/ros2_ws
   colcon build              # 这会自动编译 src/ 下的所有包，并生成 build/ 和 install/ 目录
   ```

   

3. **“激活”工作空间**：编译完成后，你必须告诉当前终端，去哪里找这些新编译的程序。这通过 `source` 命令实现。

   bash

   ```bash
   source ~/ros2_ws/install/setup.bash
   ```

   

   > **注意**：这个 `source` 命令只在当前终端窗口有效。你可以把它加到 `~/.bashrc` 文件里，这样每次打开新终端都会自动激活。

**“激活”的本质是什么呢？** 可以把它理解为**环境变量的切换**。ROS 2 系统本身也在 `/opt/ros/humble` 里，你 source 它的 `setup.bash`，就相当于告诉系统："请使用官方安装的ROS2"。而你 source 自己工作空间的 `install/setup.bash`，系统就会**优先**使用你 `install` 目录下的程序，从而覆盖或补充系统的功能。这跟你之前配置 `clangd` 时理解的环境变量 `PATH` 的原理是一样的。

### 四、一个形象的比喻

你可以把 ROS 2 的工作空间想象成一个**厨房**：

- **工作空间根目录 (`ros2_ws/`)**：整个厨房。
- **`src/` 目录**：**冰箱和储物柜**。里面放着所有的“食材”（你的源代码和从网上下载的别人的功能包）。
- **`build/` 目录**：**切菜和备菜区**。在这里把食材洗、切、腌制（编译成中间文件）。
- **`install/` 目录**：**灶台和出菜区**。在这里进行最终烹饪，做出一道道可以端上桌的“菜”（可执行文件）。
- **`source install/setup.bash` 命令**：**打开灶台的开关**。只有打开了开关，灶台（工作空间）才能工作，你才能把炒好的菜（运行节点）端上桌。
